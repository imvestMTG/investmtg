Webhook-powered scan-to-list architecture for marketplace integration.

## Architecture

Flow: `scan event` → `long-poll GET` → `enrich` → `listings`, `CSV upload`

| Component | Type | Details |
|---|---|---|
| 📱 **SELLER'S DEVICE** | CLIENT | Delver X (Pro) · Webhook enabled · Scans cards → fires POST |
| 📄 **CSV / .DLENS IMPORT** | FALLBACK | Bulk upload path · Parse .dlens (SQLite) · Parse CSV exports |
| 🔄 **DELVER WEBHOOK SERVER** | EXTERNAL | api.delver.app/webhook/{id} · Holds events until polled · 10s long-poll timeout · No HTTPS/CORS needed |
| 📡 **INVESTMTG RECEIVER** | WORKER | Polls Delver server · Maps seller → endpoint · Validates + dedupes · Pushes to job queue |
| ⏳ **JOB QUEUE** | QUEUE | Redis / BullMQ · Retry + backoff |
| 🔍 **SCRYFALL ENRICHMENT** | API | GET /cards/{scryfallId} · Hi-res images, oracle text · Legality, color identity · 50ms rate limit respected |
| ⚙️ **CARD PROCESSOR** | WORKER | Enrich via Scryfall API · Normalize condition/finish · Compute suggested price · Create draft listing |
| 🗄️ **INVESTMTG DATABASE** | POSTGRES | draft_listings table · seller_webhooks table · card_cache table |
| 🖥️ **SELLER DASHBOARD** | UI | Review draft listings · Set price / condition · Publish → live marketplace |

## Webhook Event Payload

```json
{
  "type": "card_scanned",
  "cards": [{
    "scryfallId": "fdfeeb64-0f86...",
    "name": "Birds of Paradise",
    "edition": "Secret Lair Drop",
    "editionCode": "sld",
    "price": 8.36,
    "tcgMid": 8.36,
    "mkmTrend": 9.49,
    "ckRetail": 10.99,
    "condition": "",
    "finish": "regular",
    "rarity": "R",
    "number": "92"
  }]
}
```

The `card_scanned` event contains everything needed to create a draft listing. The `scryfallId` is the bridge to enriched data (hi-res images, oracle text, legality).

### Key Fields for investMTG

- **IDENTITY:** scryfallId — primary key to Scryfall · name — card name · editionCode — set code (e.g. sld) · number — collector number
- **PRICING:** tcgMid — TCGplayer mid market · mkmTrend — Cardmarket trend · ckRetail — Card Kingdom retail · ckBuylist — Card Kingdom buylist
- **CONDITION / VARIANT:** finish — regular / foil / etched · condition — NM, LP, MP, etc. · language — card language · rarity — C, U, R, M
- **METADATA:** uuid — unique instance ID · listName — Delver list origin · quantity — count scanned · cardType — creature, instant, etc.

## Seller Flow

1. **Seller registers webhook on investMTG** — Seller links their Delver Pro account. investMTG stores their webhook endpoint URL (generated by Delver X settings). Each seller gets a unique polling worker.
2. **Seller scans cards with Delver X** — On their phone or webcam, the seller scans physical cards. Each scan fires a `card_scanned` event to `api.delver.app/webhook/{their-endpoint}`.
3. **investMTG receiver polls & ingests** — A background worker long-polls each seller's webhook endpoint. When events arrive, they're validated, deduped (by UUID), and pushed to a job queue for processing.
4. **Enrich via Scryfall + price engine** — The card processor uses the scryfallId to fetch hi-res images, oracle text, and legality. Delver's embedded prices (TCG, MKM, CK) are captured. investMTG's pricing engine computes a suggested list price.
5. **Draft listing created** — A draft_listing row is inserted with all enriched data. The seller sees it appear in real-time on their investMTG dashboard. They can adjust condition, price, and add photos.
6. **Seller reviews & publishes** — The seller reviews draft listings (individually or in bulk), confirms pricing and condition, then publishes to the live marketplace. Stripe Connect handles settlement.

## Implementation

### Recommended Stack

- **POLLING WORKER — Node.js + Bull:** One job per seller endpoint, 1s poll interval, exponential backoff on 404
- **JOB QUEUE — Redis + BullMQ:** Retry logic, rate limiting, dead letter queue for failed enrichments
- **CARD PROCESSOR — Node.js Worker:** Scryfall enrichment (50ms delay), price normalization, draft listing creation
- **DATABASE — PostgreSQL:** seller_webhooks, draft_listings, card_cache tables. JSONB for raw Delver payload
- **REALTIME UI — WebSocket / SSE:** Push new draft listings to seller dashboard as they arrive
- **FALLBACK IMPORT — File Upload API:** Parse .dlens (SQLite) and CSV. Map to same draft_listing pipeline

### Key Considerations

- **Delver Pro Requirement:** Webhooks require an active Delver Pro subscription ($4.99/mo). Without it, all cards return as 'Birds of Paradise'. Sellers must have their own Pro account.
- **Endpoint Security:** Each seller's webhook URL is effectively their auth token — the endpoint is open and secured only by its URL obscurity. Store encrypted, rotate if compromised.
- **Polling Scalability:** Each active seller needs a polling worker. For the Guam market (tens of sellers), this is trivial. At scale (1000+), consider a shared polling pool with round-robin.
- **Condition Gap:** Delver sends condition as an empty string by default — card grading must be seller-input on the investMTG side. Build condition selection into the draft review UI.

### Core Schema Sketch

```sql
-- seller_webhooks
-- id              UUID PK
-- seller_id       FK → users
-- endpoint_url    TEXT (encrypted)
-- is_active       BOOLEAN
-- last_polled_at  TIMESTAMP

-- draft_listings
-- id              UUID PK
-- seller_id       FK → users
-- scryfall_id     TEXT
-- delver_uuid     TEXT UNIQUE  -- dedup key
-- name            TEXT
-- edition_code    TEXT
-- collector_num   TEXT
-- finish          TEXT (regular/foil/etched)
-- condition       TEXT (seller-set)
-- suggested_price DECIMAL
-- seller_price    DECIMAL (nullable until set)
-- tcg_mid         DECIMAL
-- mkm_trend       DECIMAL
-- ck_retail       DECIMAL
-- status          TEXT (draft/published/sold)
-- raw_payload     JSONB
-- enriched_at     TIMESTAMP
-- created_at      TIMESTAMP
```

## Notes for investMTG Integration

- This architecture assumes Stripe Connect is live for seller payouts (settlement on publish/sale).
- The draft_listings table would sit alongside the existing `listings` table — drafts promote to listings on publish.
- Scryfall enrichment reuses the existing `/api/card/:id` proxy pattern already in the worker.
- For D1 (current stack), JSONB becomes TEXT with JSON serialization. If migrating to Postgres/Supabase later, use native JSONB.
- SOUL.md Rule 2 applies: all cards must be paper-only. Filter out any digital-only sets during enrichment.
